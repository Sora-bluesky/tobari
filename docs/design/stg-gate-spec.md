# STG Gate Specification

## 1. 設計原則

### 唯一の人間タッチポイント: STG0

| 原則                | 内容                                                      |
| ------------------- | --------------------------------------------------------- |
| **1回だけ契約**     | STG0 でユーザーと契約を結ぶ。以降は自動で前進する         |
| **tobari が翻訳**   | ユーザーは自然言語で意図を伝えるだけ。tobari が構造化する |
| **契約が帳の境界**  | 契約の範囲 = 帳の内側。範囲外の操作は fail-close          |
| **判定不能 = 停止** | 自動判定できない場合は常に fail-close（安全側に倒す）     |

### 従来モデルとの違い

```
従来: 要件定義 → [承認] → 設計 → [承認] → 実装 → [承認] → テスト → [承認]
                 ↑毎回の承認 → 承認疲れ → 全部OKで押す → 形骸化

tobari: 意図 → [契約(1回)] → 設計 → 実装 → テスト → 完了
               ↑帳をおろす   ↑以降は自動。異常時のみ fail-close で停止
```

### ゲート遷移の自動判定

```
STG{N} の DoD を全て充足
  → Evidence Ledger に証跡を記録
  → tobari-session.json の gates_passed を更新
  → STG{N+1} に自動遷移

STG{N} の DoD に未達項目あり
  → 未達理由を分析
  → 自動修復を試行（リトライ上限あり）
  → 修復不能 → fail-close（理由 + 復旧手順を出力）
```

## 2. STG0: 帳をおろす儀式（Requirements）

### 2.1 STG0 が特別な理由

STG0 は唯一の人間タッチポイントであり、**帳の境界を定義する儀式**。
ここで結ばれる契約の質が、以降のすべてを決める。

初心者は構造的な指示ができない。だから tobari が要件翻訳者になる。

### 2.2 フロー

```
ユーザー入力（自然言語）
  │
  ▼
① 意図の受取（receive_intent）
  │  入力: 自由テキスト（構造不要）
  │  例: 「ログイン機能を追加して」
  │
  ▼
② コンテキスト分析（analyze_context）
  │  - コードベース構造を分析（Claude Code + サブエージェント）
  │  - 影響範囲を特定
  │  - リスクレベルを評価
  │
  ▼
③ 契約の生成（generate_contract）
  │  - やること / やらないこと
  │  - 完了基準（DoD チェックリスト）
  │  - 影響ファイル・ディレクトリ（スコープ）
  │  - リスクレベル → Profile 自動選択
  │
  ▼
④ ユーザー確認（user_confirm）
  │  「この内容で帳をおろして実装を開始しますか？」
  │  → YES: ⑤ へ
  │  → 修正: ③ に戻る（修正ループ）
  │  → キャンセル: 中止
  │
  ▼
⑤ 帳をおろす（lower_veil）
     - tobari-session.json を作成
     - ブロッキング Hooks を有効化
     - Evidence Ledger を開始
     - 詠唱 + ステータス表示
```

### 2.3 契約の構造

tobari-session.json に記録される契約:

```json
{
  "active": true,
  "task": "add-login",
  "profile": "standard",
  "started_at": "2026-02-25T14:00:00Z",
  "gates_passed": ["STG0"],
  "contract": {
    "intent": "ログイン機能を追加して",
    "requirements": {
      "do": [
        "メールアドレス+パスワードでログインできる画面を作る",
        "ログイン状態を保持する"
      ],
      "do_not": ["パスワードリセット機能", "ソーシャルログイン"]
    },
    "dod": [
      "ログイン画面が表示される",
      "正しい情報でログインできる",
      "間違った情報でエラーメッセージが出る",
      "テストが通る"
    ],
    "scope": {
      "include": ["src/pages/login/", "src/api/auth/"],
      "exclude": ["src/pages/admin/", "src/api/payment/"]
    },
    "risk_level": "medium"
  }
}
```

### 2.4 初心者が構造的指示をできない問題への対応

| 問題                       | tobari の対応                                          |
| -------------------------- | ------------------------------------------------------ |
| 何を作るか言語化できない   | tobari が候補を提案（「こういうことですか？」）        |
| やらないことを想定できない | テンプレートから典型的な除外項目を自動提示             |
| 完了基準が書けない         | コードベース分析 + パターンから DoD を自動生成         |
| 影響範囲がわからない       | Claude Code でコードベースを分析し、スコープを自動特定 |
| 確認内容が理解できない     | 技術用語を使わず結果ベースの日本語で提示               |

### 2.5 確認 UX の例

```
📋 帳の契約

やること:
  ✓ メールアドレスとパスワードでログインできる画面を作る
  ✓ ログイン状態を保持する

やらないこと:
  ✗ パスワードリセット機能
  ✗ ソーシャルログイン（Google, GitHub 等）

できあがりの確認項目:
  □ ログイン画面が表示される
  □ 正しい情報でログインできる
  □ 間違った情報でエラーメッセージが出る
  □ テストが通る

影響する場所:
  src/pages/login/  src/api/auth/

リスク: 中 → Standard プロファイル

この内容で帳をおろして実装を開始しますか？ [はい / 修正する / やめる]
```

### 2.6 DoD

| #   | 条件                                           | 判定方法                            |
| --- | ---------------------------------------------- | ----------------------------------- |
| 1   | ユーザーの意図が自然言語で受け取れた           | 入力の存在確認                      |
| 2   | 契約（requirements + dod + scope）が生成された | 契約フィールドの充足確認            |
| 3   | ユーザーが「はい」で承認した                   | user_confirm = yes                  |
| 4   | tobari-session.json が作成された               | ファイル存在 + スキーマ検証         |
| 5   | Profile が自動選択された                       | risk_level → profile マッピング確認 |

### 2.7 fail-close 条件

| 条件                                   | 対応                                       |
| -------------------------------------- | ------------------------------------------ |
| 意図が空または解釈不能                 | 具体例を示して再入力を促す                 |
| コードベース分析に失敗                 | エラー内容を提示し、手動スコープ指定を促す |
| ユーザーが「やめる」を選択             | 帳をおろさずに終了                         |
| リスクが high かつ Strict プロファイル | 追加確認を要求（Strict のみの例外）        |

## 3. STG1: Design（設計）

### 3.1 目的

契約に基づいて実装アプローチを決定する。

### 3.2 自動判定の流れ

```
契約の requirements を読み取る
  → コードベースの既存パターンを分析
  → 実装アプローチを決定
  → 設計決定を Evidence Ledger に記録
  → DoD 充足 → STG2 へ自動遷移
```

### 3.3 DoD

| #   | 条件                             | 判定方法                           |
| --- | -------------------------------- | ---------------------------------- |
| 1   | 実装アプローチが決定されている   | evidence に設計ログが存在          |
| 2   | 影響ファイル一覧が特定されている | scope が具体的なファイルパスに解決 |
| 3   | 既存コードとの競合がない         | 境界分類との整合確認               |

### 3.4 fail-close 条件

| 条件                                         | 対応                                    |
| -------------------------------------------- | --------------------------------------- |
| requirements 間に矛盾を検出                  | 矛盾内容を提示し、STG0 の契約修正を提案 |
| スコープ内に境界分類 UNCLASSIFIED のファイル | 分類を先に解決するよう停止              |
| 実行可能なアプローチが見つからない           | 技術的制約を提示し、契約の修正を提案    |

### 3.5 Lite プロファイルでの扱い

Lite ではスキップ。STG0 の契約生成時に簡易設計を含める。

## 4. STG2: Implementation（実装）

### 4.1 目的

契約の requirements に基づいてコードを書く。

### 4.2 自動判定の流れ

```
STG1 の設計に基づいて実装
  → 契約の scope 内のファイルのみ変更
  → 自己レビュー（コード品質・セキュリティ）
  → DoD 充足 → STG3 へ自動遷移
```

### 4.3 DoD

| #   | 条件                                       | 判定方法                                             |
| --- | ------------------------------------------ | ---------------------------------------------------- |
| 1   | 契約の requirements がすべて実装されている | requirements.do の各項目に対応するコード変更が存在   |
| 2   | do_not に含まれる機能が実装されていない    | 除外項目に関連するコードが追加されていないことを確認 |
| 3   | 変更が scope 内に収まっている              | 変更ファイル一覧 ⊆ contract.scope.include            |
| 4   | 自己レビューが完了している                 | Evidence Ledger に自己レビューログが存在             |

### 4.4 fail-close 条件

| 条件                                          | 対応                                               |
| --------------------------------------------- | -------------------------------------------------- |
| scope 外のファイルを変更しようとした          | **帳が止める**: 「契約範囲外の操作を検出」と表示   |
| do_not に含まれる機能を実装しようとした       | **帳が止める**: 「契約で除外された機能です」と表示 |
| 秘密情報（パスワード、APIキー）のハードコード | **帳が止める**: git-guard 連携でブロック           |
| 破壊的操作（ファイル全削除等）                | **帳が止める**: ブロッキング Hook で自動停止       |

### 4.5 スコープ逸脱時の UX

```
🔒 帳が止めました

理由: 契約範囲外の操作を検出
  契約: 「ログイン機能の追加」
  検出: src/pages/admin/settings.tsx への変更

選択肢:
  1. この操作をスキップして続行する
  2. 契約を更新する（帳をおろし直す → STG0 に戻る）
  3. 作業を中断する
```

## 5. STG3: Verification（検証）

### 5.1 目的

実装が正しく動作することを自動検証する。

### 5.2 自動判定の流れ

```
テストを実行
  → lint / formatter チェック
  → 契約の DoD チェックリストを照合
  → DoD 充足 → STG4 へ自動遷移
```

### 5.3 DoD

| #   | 条件                                | 判定方法                          |
| --- | ----------------------------------- | --------------------------------- |
| 1   | テストが全て通る                    | テストランナーの exit code = 0    |
| 2   | lint エラーがない                   | linter の exit code = 0           |
| 3   | 契約の dod チェックリストが全て充足 | 各 dod 項目に対する検証証跡が存在 |

### 5.4 fail-close 条件

| 条件               | 対応                                        |
| ------------------ | ------------------------------------------- |
| テスト失敗         | 自動修復を試行（最大3回）。修復不能なら停止 |
| lint エラー        | 自動修正を試行。修正不能なら停止            |
| DoD 項目の検証不能 | 検証方法がない項目を提示し、手動確認を要求  |

### 5.5 エージェントの自己申告を信用しない

> **重要**: STG3 はエージェントの「完了しました」を信用しない。

テスト結果・lint 結果は**実際のコマンド出力**で判定する。
エージェントが「テスト通りました」と言っても、実行ログがなければ DoD 未達。

## 6. STG4: Automation（CI/CD）

### 6.1 目的

CI/CD パイプラインによる自動チェックを通す。

### 6.2 自動判定の流れ

```
CI/CD チェックを実行（GitHub Actions 等）
  → 全チェック green
  → DoD 充足 → STG5 へ自動遷移
```

### 6.3 DoD

| #   | 条件                         | 判定方法                              |
| --- | ---------------------------- | ------------------------------------- |
| 1   | 必須 CI チェックが全て green | GitHub Actions のステータス確認       |
| 2   | セキュリティスキャンが通る   | git-guard + CI のセキュリティチェック |

### 6.4 fail-close 条件

| 条件                     | 対応                             |
| ------------------------ | -------------------------------- |
| CI チェック失敗          | 失敗内容を分析し、自動修復を試行 |
| セキュリティスキャン検出 | 即座に停止。検出内容を提示       |

### 6.5 CI 未構成の場合

CI が設定されていないプロジェクトでは、STG3 の検証結果をもって STG4 を自動通過とする。
ただし Evidence Ledger に「CI 未構成のため STG3 結果で代替」と記録する。

## 7. STG5: Commit/Push

### 7.1 目的

変更をバージョン管理に記録し、リモートに送る。

### 7.2 自動判定の流れ

```
変更をステージング
  → git-guard（pre-commit hook）チェック
  → コミット作成
  → git-guard（pre-push hook）チェック
  → リモートに push
  → DoD 充足 → STG6 へ自動遷移
```

### 7.3 DoD

| #   | 条件                       | 判定方法                           |
| --- | -------------------------- | ---------------------------------- |
| 1   | 変更がコミットされている   | git log でコミット存在確認         |
| 2   | git-guard チェックが通る   | pre-commit / pre-push hook の PASS |
| 3   | リモートに push されている | git push の成功確認                |

### 7.4 fail-close 条件

| 条件                       | 対応                                   |
| -------------------------- | -------------------------------------- |
| git-guard が秘密情報を検出 | **即停止**。検出ファイル・行を提示     |
| プライベートパスの検出     | **即停止**。匿名化を実行してから再試行 |
| push 先が protected branch | feature branch に切り替えて push       |

## 8. STG6: PR/Merge

### 8.1 目的

Pull Request を作成し、マージする。

### 8.2 自動判定の流れ

```
PR を作成（タイトル・本文は契約から自動生成）
  → CI チェック完了を待機
  → [Strict のみ] 人間レビューを待機
  → マージ
  → DoD 充足 → 帳を解除
```

### 8.3 DoD

| #   | 条件                              | 判定方法                          |
| --- | --------------------------------- | --------------------------------- |
| 1   | PR が作成されている               | gh pr view で存在確認             |
| 2   | CI チェックが全て green           | PR のステータスチェック           |
| 3   | [Standard] blocking 指摘が 0      | quality_gate_counts.blocking == 0 |
| 4   | [Strict] blocking + high 指摘が 0 | blocking == 0 かつ high == 0      |
| 5   | マージが完了している              | PR のマージステータス確認         |

### 8.4 fail-close 条件

| 条件                        | 対応                               |
| --------------------------- | ---------------------------------- |
| CI チェック失敗             | STG4 に戻って再修復                |
| blocking レビュー指摘       | STG2 に戻って修正                  |
| [Strict] 人間レビュー未完了 | レビュー待ちで停止（自動催促あり） |

### 8.5 帳の解除

STG6 完了時:

```
帳の内側での作業が完了しました。

📋 証跡サマリー:
  - 変更ファイル: 8
  - テスト: 12 passed
  - CI: all green
  - 所要時間: 45分

帳を解除します。
```

tobari-session.json を削除（= 帳を解除）。Evidence Ledger を close。

## 9. プロファイル別ゲート適用サマリー

| Gate     | Lite       | Standard            | Strict                        |
| -------- | ---------- | ------------------- | ----------------------------- |
| **STG0** | 簡易契約   | 完全契約            | 完全契約 + 追加確認           |
| STG1     | スキップ   | 自動                | 自動                          |
| STG2     | 自動       | 自動 + スコープ監視 | 自動 + スコープ監視           |
| STG3     | 基本テスト | 全テスト + lint     | 全テスト + lint + カバレッジ  |
| STG4     | スキップ   | CI 必須             | CI + セキュリティスキャン必須 |
| STG5     | 自動       | 自動                | 自動                          |
| **STG6** | 自動マージ | 自動マージ          | **人間レビュー必須**          |
